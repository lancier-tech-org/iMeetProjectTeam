## Platform Overview

iMeetPro Media Server is a self-hosted WebRTC media infrastructure built from LiveKit's open-source code and Coturn TURN/STUN server. It replaces LiveKit's cloud service with a fully self-hosted solution running on your own hardware. Zero data leaves your infrastructure — all media routing, signaling, and NAT traversal happen on your servers.

The system is composed of **3 independently containerized services** — an SFU (Selective Forwarding Unit) with WebSocket signaling, a TURN/STUN server for NAT traversal, and an Nginx reverse proxy for SSL termination.

**What this server does:**
- Receives audio/video streams from each meeting participant via WebRTC
- Selectively forwards streams to other participants in the same room (SFU architecture — no transcoding, no mixing)
- Handles WebSocket signaling for room join/leave, track publish/subscribe, SDP offer/answer exchange, and ICE candidate exchange
- Provides TURN/STUN relay for participants behind restrictive NATs or firewalls
- Terminates SSL via Nginx so browsers can connect securely over WSS

**What this server does NOT do:**
- Store any data (no database, no persistent storage)
- Handle authentication (tokens are generated by the iMeetPro backend)
- Record meetings (the recording bot in the backend connects as a participant)
- Manage users, rooms in a database, or any application logic

## Architecture

```
                     ┌─────────────────────────────┐
                     │     Client Browsers          │
                     │  (iMeetPro Frontend App)     │
                     └──────────┬──────────────────┘
                                │
                                │ wss://host:8443 (WebSocket Signaling)
                                │ UDP 50000-60000 (Media Streams)
                                │ UDP 5349 (TURN/STUN)
                                │
                     ┌──────────▼──────────────────┐
                     │   Nginx SSL Proxy (:8443)    │
                     │   SSL Termination            │
                     │   WebSocket Upgrade           │
                     └──────────┬──────────────────┘
                                │
                                │ http://127.0.0.1:7880
                                │
                     ┌──────────▼──────────────────┐
                     │   iMeetPro Media Server      │
                     │   (SFU + Signaling)          │
                     │   :7880 (HTTP/WS)            │
                     │   :7881 (RTC TCP)            │
                     │   :50000-60000 (Media UDP)   │
                     └──────────────────────────────┘

                     ┌──────────────────────────────┐
                     │   Coturn TURN/STUN Server     │
                     │   :5349 (UDP+TCP)             │
                     │   :50000-60000 (Relay UDP)    │
                     └──────────────────────────────┘
```

**How a meeting connection works:**

1. User clicks "Join Meeting" in iMeetPro frontend
2. Frontend calls backend API (`/api/livekit/join-meeting/`) to get an access token
3. Backend generates a JWT token using the API key and secret, returns it with the WSS URL
4. Frontend's LiveKit client SDK connects to `wss://host:8443/rtc?access_token=...`
5. Nginx terminates SSL and proxies to `http://127.0.0.1:7880`
6. Media server validates the token, creates/joins the room, negotiates WebRTC connection
7. Client asks Coturn STUN server "what is my public IP?" for ICE candidates
8. WebRTC media connection is established directly to the SFU (or via TURN relay if needed)
9. Audio/video RTP packets flow between client and SFU on UDP ports 50000-60000
10. SFU forwards each participant's streams to all other participants in the room

**Inter-service communication with iMeetPro backend:**

- **Backend → Media Server**: Room creation, participant listing, participant removal, track muting (via Twirp RPC API on port 7880, proxied through Nginx on 8443)
- **Frontend → Media Server**: WebSocket signaling + WebRTC media (via Nginx SSL proxy on 8443)
- **Recording Bot → Media Server**: Joins as a hidden participant via LiveKit Python SDK, receives all tracks for recording

## Service Descriptions

### 1. iMeetPro Media Server (SFU + Signaling)

Built from LiveKit open-source Go code (`github.com/livekit/livekit` v1.10.1). Compiled into a single binary `imeetpro-server`.

**Responsibilities:**
- WebSocket signaling server on port 7880 — handles room join/leave, SDP negotiation, ICE candidate exchange, track publish/subscribe events
- SFU (Selective Forwarding Unit) — receives audio/video RTP packets from each participant and selectively forwards to others in the same room. No transcoding or mixing.
- Room management — auto-creates rooms, tracks participants, enforces permissions from JWT tokens
- Twirp RPC API — exposes `livekit.RoomService` endpoints for backend to create rooms, list participants, remove participants, mute tracks
- Built-in TURN integration — coordinates with external Coturn server for ICE candidates

**Key configuration (`config/config.yaml`):**
- `port: 7880` — signaling HTTP/WebSocket port
- `rtc.tcp_port: 7881` — TCP fallback for WebRTC
- `rtc.port_range_start/end: 50000-60000` — UDP port range for media
- `rtc.node_ip` — public IP address of the server (MUST match the server's actual IP)
- `keys` — API key and secret pair for JWT token validation
- `room.max_participants: 0` — 0 means unlimited, limited only by hardware
- `turn.udp_port: 5349` — tells clients where to find the TURN server

**Source code:** `livekit/` directory (Go)

**Dependencies compiled in automatically:**
- `github.com/livekit/protocol` — protobuf message definitions
- `github.com/livekit/mediatransportutil` — WebRTC media transport utilities
- `github.com/pion/webrtc` — Go WebRTC implementation

### 2. Coturn TURN/STUN Server

Built from Coturn open-source C code (`github.com/coturn/coturn` v4.9.0).

**Responsibilities:**
- **STUN**: Helps clients discover their public IP and port for WebRTC ICE candidates. Works for ~80-85% of users.
- **TURN**: Relays media packets for clients behind restrictive NATs or firewalls (corporate networks, hotel WiFi, some mobile networks). Acts as a middleman: Client → TURN → SFU and back. Slower but guarantees connectivity.

**Key configuration (`config/turnserver.conf`):**
- `listening-port=5349` — STUN/TURN listener
- `listening-ip` — server's IP address
- `relay-ip` — IP used for relaying (usually same as listening-ip)
- `external-ip` — public IP (for AWS, set to public IP / private IP)
- `min-port/max-port: 50000-60000` — UDP relay port range
- `user=imeetpro:imeetpro2026` — TURN authentication credentials
- `realm=imeetpro` — authentication realm

**Source code:** `coturn/` directory (C)

### 3. Nginx SSL Proxy

Uses official `nginx:alpine` image — no custom build needed.

**Responsibilities:**
- SSL/TLS termination on port 8443 using server certificate and key
- WebSocket upgrade proxy (`Upgrade: websocket`, `Connection: upgrade`)
- Forwards all traffic to `http://127.0.0.1:7880` (the SFU signaling port)
- Provides the `wss://` endpoint that browsers require for secure WebSocket connections

**Key configuration (`nginx/nginx.conf`):**
- `listen 8443 ssl http2` — HTTPS listener
- `proxy_pass http://127.0.0.1:7880` — upstream to SFU
- `proxy_read_timeout 86400` — 24-hour timeout for long-lived WebSocket connections
- SSL certificate and key in `nginx/certs/`

## Environment Configuration

### config/config.yaml (SFU + Signaling)

```yaml
port: 7880
bind_addresses:
  - "0.0.0.0"
rtc:
  port_range_start: 50000
  port_range_end: 60000
  tcp_port: 7881
  node_ip: <SERVER_PUBLIC_IP>       # ⚠️  CHANGE THIS — must be server's actual IP
  use_external_ip: false            # Set true on AWS if behind NAT
keys:
  imeetpro_key: <API_SECRET>        # ⚠️  CHANGE THIS — must match backend .env
room:
  auto_create: true
  max_participants: 0               # 0 = unlimited
turn:
  enabled: true
  domain: imeetpro
  tls_port: 0
  udp_port: 5349
  external_tls: false
log_level: info
```

### config/turnserver.conf (TURN/STUN)

```
listening-port=5349
listening-ip=<SERVER_PUBLIC_IP>     # ⚠️  CHANGE THIS
relay-ip=<SERVER_PUBLIC_IP>         # ⚠️  CHANGE THIS
external-ip=<SERVER_PUBLIC_IP>      # ⚠️  CHANGE THIS (on AWS: public-ip/private-ip)
min-port=50000
max-port=60000
fingerprint
lt-cred-mech
user=imeetpro:imeetpro2026
realm=imeetpro
no-tls
no-dtls
log-file=/tmp/turnserver.log
simple-log
```

### nginx/nginx.conf (SSL Proxy)

```nginx
server {
    listen 8443 ssl http2;
    server_name _;

    ssl_certificate /etc/nginx/certs/server.crt;
    ssl_certificate_key /etc/nginx/certs/server.key;

    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers ECDHE-RSA-AES128-GCM-SHA256:ECDHE-RSA-AES256-GCM-SHA384;

    location / {
        proxy_pass http://127.0.0.1:7880;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_read_timeout 86400;
        proxy_buffering off;
    }
}
```

### SSL Certificates

Place your SSL certificate files in `nginx/certs/`:

```
nginx/certs/server.crt    — SSL certificate (PEM format)
nginx/certs/server.key    — SSL private key (PEM format)
```

**Development:** Use self-signed certificates (users must accept browser warning once).

**Production (AWS):** Use Let's Encrypt with a domain name:
```bash
sudo certbot certonly --standalone -d media.yourdomain.com
cp /etc/letsencrypt/live/media.yourdomain.com/fullchain.pem nginx/certs/server.crt
cp /etc/letsencrypt/live/media.yourdomain.com/privkey.pem nginx/certs/server.key
```

## AWS Production Deployment

### Recommended Instance

| Component | Instance Type | vCPUs | RAM | Network | Cost/Month |
|-----------|--------------|-------|-----|---------|-----------|
| Media Server | `c6i.8xlarge` | 32 | 64 GB | 12.5 Gbps | ~$980 |

For massive scale (1000+ concurrent video participants): `c6i.16xlarge` (64 vCPUs, 128 GB RAM, 25 Gbps).

### Step-by-Step AWS Setup

**1. Launch EC2 Instance**
- AMI: Ubuntu 24.04 LTS
- Instance type: `c6i.8xlarge` (or larger)
- Storage: 100 GB gp3
- Elastic IP: Allocate and associate
- Security Group: Open ports as listed in Network section above

**2. Install Docker**
```bash
sudo apt update
sudo apt install -y docker.io docker-compose-v2
sudo usermod -aG docker $USER
# Log out and back in
```

**3. Upload Code**
```bash
# From your local machine
scp -r imeetpro-media-server/ ubuntu@<AWS_IP>:~/
```

**4. Configure for AWS**
```bash
cd ~/imeetpro-media-server

# Get the EC2 public IP
PUBLIC_IP=$(curl -s http://169.254.169.254/latest/meta-data/public-ipv4)
PRIVATE_IP=$(curl -s http://169.254.169.254/latest/meta-data/local-ipv4)

# Update configs with actual IPs
sed -i "s/192.168.48.201/$PRIVATE_IP/g" config/config.yaml
sed -i "s/use_external_ip: false/use_external_ip: true/g" config/config.yaml

sed -i "s/192.168.48.201/$PRIVATE_IP/g" config/turnserver.conf
# For TURN, set external-ip as public/private mapping
sed -i "s/^external-ip=.*/external-ip=$PUBLIC_IP\/$PRIVATE_IP/" config/turnserver.conf
```

**5. SSL with Let's Encrypt (requires domain)**
```bash
sudo apt install -y certbot
sudo certbot certonly --standalone -d media.yourdomain.com

cp /etc/letsencrypt/live/media.yourdomain.com/fullchain.pem nginx/certs/server.crt
cp /etc/letsencrypt/live/media.yourdomain.com/privkey.pem nginx/certs/server.key
```

**6. Build and Start**
```bash
docker-compose up --build -d
docker ps
```

**7. Update iMeetPro Backend**

In your backend `.env`:
```env
LIVEKIT_URL=wss://media.yourdomain.com:8443
LIVEKIT_API_KEY=imeetpro_key
LIVEKIT_API_SECRET=imeetpro_secret_2026_production_secure_key_x
```

In your frontend `constants.js`:
```javascript
url: 'wss://media.yourdomain.com:8443',
apiKey: 'imeetpro_key',
apiSecret: 'imeetpro_secret_2026_production_secure_key_x',
```

Rebuild backend: `docker-compose up --build -d`

## Performance & Capacity

### Hardware Requirements

| Participants per Room | Recommended vCPUs | RAM | Network |
|----------------------|-------------------|-----|---------|
| Up to 50 | 8 | 16 GB | 1 Gbps |
| Up to 200 | 16 | 32 GB | 5 Gbps |
| Up to 500 | 32 | 64 GB | 12.5 Gbps |
| 500+ | 64 | 128 GB | 25 Gbps |

### Bandwidth Calculation

- Each video participant: ~1-3 Mbps (send + receive)
- Each audio-only participant: ~50-100 Kbps
- **Formula:** Total bandwidth ≈ (video_participants × 2 Mbps) + (audio_participants × 75 Kbps)
- **Example:** 200 video participants ≈ 400 Mbps, 1000 audio-only ≈ 75 Mbps

### Scaling Notes

- `max_participants: 0` in config means no software limit — hardware is the only constraint
- Multiple rooms share the same server resources
- The SFU does NOT transcode — it only forwards packets, so CPU usage scales linearly with participant count
- GPU is NOT used by the media server (GPU is used by your recording bot in the backend)
- For horizontal scaling (multiple SFU instances), add Redis and configure `redisrouter` in the LiveKit config

### Backend Services That Communicate with Media Server

| Backend Service | Communication | Purpose |
|----------------|---------------|---------|
| `meeting-core-service` | Twirp RPC via HTTPS | Create rooms, list participants, remove participants, mute tracks, generate tokens |
| `recording-service` | LiveKit Python SDK via WSS | Recording bot joins rooms, subscribes to tracks, records audio/video |

**meeting-core-service/.env:**
```env

```

**recording-service/.env:**
```env
LIVEKIT_URL=wss://<MEDIA_SERVER_IP>:8443
LIVEKIT_API_KEY=imeetpro_key
LIVEKIT_API_SECRET=imeetpro_secret_2026_production_secure_key_x
```

### Frontend Configuration

**src/utils/constants.js:**
```javascript
export const LIVEKIT_CONFIG = {
  url: 'wss://<MEDIA_SERVER_IP>:8443',
  apiKey: 'imeetpro_key',
  apiSecret: 'imeetpro_secret_2026_production_secure_key_x',
};
```

### Token Flow

1. User clicks "Join Meeting" → Frontend calls backend `/api/livekit/join-meeting/`
2. Backend's `meetings.py` uses `LIVEKIT_API_KEY` and `LIVEKIT_API_SECRET` to generate a JWT token
3. Backend also calls Media Server's Twirp API to create the room if it doesn't exist
4. Backend returns `{ access_token, room_name, livekit_url }` to frontend
5. Frontend's LiveKit JS SDK connects to `livekit_url` with the `access_token`
6. Media Server validates the JWT signature using the same key/secret pair

**Critical: The `keys` in `config/config.yaml` MUST match the `LIVEKIT_API_KEY` and `LIVEKIT_API_SECRET` in the backend `.env` files. If they don't match, token validation will fail and no one can join meetings.**

## Troubleshooting

### Container won't start

```bash
# Check logs
docker-compose logs <service-name>

# Common causes:
# - Port already in use: sudo lsof -i :<port> then kill the process
# - SSL cert missing: ensure nginx/certs/server.crt and server.key exist
# - Wrong IP in config: verify node_ip matches actual server IP
```

### "Address already in use" error

Previous instance is still running on the port:
```bash
# Find and kill the process
sudo fuser -k <port>/tcp
sudo fuser -k <port>/udp

# Restart the container
docker-compose restart <service-name>
```

### "ERR_CERT_AUTHORITY_INVALID" in browser

Self-signed certificate not trusted. Open `https://<server-ip>:8443` directly in a new tab, click "Advanced" → "Proceed". This is a development-only issue — use Let's Encrypt in production.

### "Mixed Content" / "insecure WebSocket" error

Frontend is on HTTPS but trying to connect via `ws://` instead of `wss://`. Ensure:
- Frontend config uses `wss://` (not `ws://`)
- Backend `.env` uses `wss://` (not `ws://`)
- Nginx is running and listening on port 8443

### Coturn "Cannot bind" error

Port 5349 is in use by another process or a previous Coturn instance:
```bash
sudo fuser -k 5349/udp
sudo fuser -k 5349/tcp
docker-compose restart coturn
```

### Nginx "bind() failed: Address in use"

Another Nginx or process is using port 8443:
```bash
sudo systemctl stop nginx    # Stop system Nginx if running
sudo fuser -k 8443/tcp
docker-compose restart nginx
```

### Frontend can't connect but backend API works

- Check that `config.yaml` `node_ip` is correct
- Check that UDP ports 50000-60000 are open in firewall/security group
- Check that TURN server is running: `docker ps | grep coturn`
- Test STUN: browser DevTools → WebRTC Internals

### No audio/video but signaling works

- UDP ports 50000-60000 are likely blocked
- TURN server may not be running — check `docker logs imeetpro-coturn`
- On AWS, verify security group allows UDP 50000-60000 from 0.0.0.0/0

### Backend can't create rooms or list participants

- Verify backend can reach `https://<MEDIA_SERVER_IP>:8443`
- Verify API key/secret in backend `.env` matches `keys` in `config/config.yaml`
- Check media server logs: `docker-compose logs -f imeetpro-server | grep twirp`

## Phase 2 Roadmap

The current deployment uses LiveKit's open-source code running on your own servers. Phase 2 involves gradually replacing LiveKit code with your own custom implementation:

| Phase | Component | Replace With | Status |
|-------|-----------|-------------|--------|
| Phase 1 | LiveKit server binary | Self-hosted LiveKit open source | ✅ Complete |
| Phase 2a | Signaling protocol | Custom WebSocket signaling server | Planned |
| Phase 2b | SFU media routing | Custom SFU using Pion WebRTC (Go) | Planned |
| Phase 2c | Frontend SDK | Custom WebRTC client using browser APIs | Planned |
| Phase 2d | Backend SDK | Custom Python token generator + API client | Planned |
| Phase 2e | TURN/STUN | Keep Coturn (industry standard, no need to replace) | — |

**Phase 2 goal:** Zero LiveKit code anywhere in the stack. Full ownership of every component.
